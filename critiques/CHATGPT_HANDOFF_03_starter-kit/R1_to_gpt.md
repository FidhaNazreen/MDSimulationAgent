# Round 1 handoff: standalone starter-kit for the mdagent MD pipeline

---

## Section 1 — Context bundle

### What's already built

`mdagent` ships as an installable Python tool (`uv tool install
git+https://github.com/FidhaNazreen/MDSimulationAgent@v0.1.0`) with three
Claude skills packaged inside the wheel: `md-run-workflow`,
`md-prep-structure`, `md-visualize`. The CLI exposes:

```
mdagent --version
mdagent doctor [--json] [--min-version] [--gmx-required] [--check-network] [--check-viewers]
mdagent run-workflow --runs-root DIR --pdb-id ID|--structure-path PATH [--config CFG] ...
mdagent prep-structure --runs-root DIR --pdb-id ID|--structure-path PATH ...
mdagent visualize --run-root DIR ...
mdagent inspect --run-root DIR
mdagent install-skills [--user | --project DIR] [--dry-run]
mdagent self-test resources [--json]
```

A user who has run `mdagent install-skills --user` ALREADY has the
skills discoverable everywhere — Claude Code finds them in
`~/.claude/skills/`. So in some sense the "starter kit" is empty: you
just install mdagent and go.

### What the user actually wants

A standalone folder that's the basis for **a new repository**. Someone
new to this project should be able to clone or copy that folder, run
one command, and have a working MD-simulation environment with a clear
tutorial. The folder is the **scaffold for a new MD project**, not a
new copy of the source code.

### What I'm proposing

Add a `starter_kit/` directory inside the package
(`src/mdagent/_resources/starter_kit/`) plus a new CLI subcommand
`mdagent init-project DIR` that materializes it into a target directory.
The materialized layout:

```
<DIR>/
├── README.md                       # Quick start; install + first run
├── .claude/skills/
│   ├── md-run-workflow/SKILL.md    # (copied from package)
│   ├── md-prep-structure/SKILL.md
│   └── md-visualize/SKILL.md
├── run_configs/
│   ├── lysozyme_tutorial.json      # Reproduces the canonical tutorial
│   ├── lysozyme_short.json         # 100 ps each phase, no production (~2 min)
│   └── general_md_prep_example.json # general mode with -inter
├── tutorial/
│   └── getting_started.ipynb       # < 10 cells; install + golden path + reading results
├── runs/                           # output dir (empty; created lazily)
│   └── .gitkeep
├── .gitignore                      # ignores runs/* but keeps .gitkeep
└── verify.sh                       # one-shot "does my install work?" check
```

Then in any new repo:

```bash
mdagent init-project ./my-md-study
cd my-md-study
./verify.sh    # → confirms mdagent + gmx + skill install
mdagent run-workflow --runs-root ./runs --config ./run_configs/lysozyme_short.json --run-id demo
mdagent inspect --run-root ./runs/demo
```

### Required code changes

  - New `src/mdagent/_resources/starter_kit/` with all the templated
    files. README + tutorial reference `mdagent` only, no `cd <path>`.
  - New CLI subcommand `mdagent init-project DIR`:
    - Refuses to overwrite an existing non-empty directory unless
      `--force`.
    - Materializes the starter kit tree.
    - Then calls `mdagent install-skills --project DIR` internally so
      the skills are immediately wired.
    - Prints a JSON summary of what was written.
  - `verify.sh` is a thin wrapper around `mdagent doctor --gmx-required
    --check-network` plus `mdagent self-test resources --json`. Exit
    non-zero if anything fails so users get a deterministic signal.
  - Tutorial notebook is < 10 cells: install, write a config, run the
    workflow, inspect, read analysis.json. Generated by a small
    `build_starter_tutorial.py` (kept in `tutorial/` of THIS repo, not
    in the starter kit).

### What I'm NOT proposing

  - **Vendoring mdagent source into the starter kit.** Users install
    mdagent once globally; the starter kit references it by name. That
    keeps the kit tiny.
  - **A separate git repo for the starter kit.** Ships inside this
    repo, materialized via the CLI on demand.
  - **Custom build steps in the starter kit.** No `pyproject.toml`, no
    `setup.py`, no `uv` dependency on the user's side beyond what
    mdagent itself already requires.

### Verification plan (after the design converges)

  - Copy the materialized kit to `/tmp/mdagent_starter_test/`.
  - From that directory (no PYTHONPATH, no checkout of THIS repo), run:
    ```
    ./verify.sh
    mdagent run-workflow --runs-root ./runs \
      --config ./run_configs/lysozyme_short.json --run-id verify
    ```
  - Assert `./runs/verify/REPORT.md` exists with `readiness: ready`.
  - Assert `./runs/verify/step_10_analysis/analysis.json` has the
    expected Rg ≈ 1.4 nm summary.

## Section 2 — Artifact under review

The plan I want you to pressure-test before I code it.

### 2.1 `mdagent init-project` contract

```
mdagent init-project [--force] [--no-install-skills] [--json] DIR
```

  - `DIR`: target directory. Created if it doesn't exist.
  - `--force`: overwrite existing files in DIR (not used by default
    — refuse to clobber if DIR exists and is non-empty).
  - `--no-install-skills`: skip the implicit
    `mdagent install-skills --project DIR` call (useful for users who
    want `--user` skills instead).
  - `--json`: machine-readable summary on stdout.

Materialized files (verbatim copies from
`src/mdagent/_resources/starter_kit/`, except `.claude/skills/` which
is built by the install-skills call):

```
DIR/README.md
DIR/run_configs/lysozyme_tutorial.json
DIR/run_configs/lysozyme_short.json
DIR/run_configs/general_md_prep_example.json
DIR/tutorial/getting_started.ipynb
DIR/runs/.gitkeep
DIR/.gitignore
DIR/verify.sh
DIR/.claude/skills/<3 skills>  # via mdagent install-skills --project DIR
```

### 2.2 README.md skeleton

```markdown
# my-md-study — agent-driven MD simulation

This project uses **mdagent** + Claude skills to run GROMACS MD
simulations from a natural-language prompt.

## Setup (one-time per machine)

(if you haven't already done this)
…install uv, mdagent, gmx as in the upstream mdagent README…

## First run

./verify.sh

mdagent run-workflow --runs-root ./runs \
  --config ./run_configs/lysozyme_short.json --run-id demo

mdagent inspect --run-root ./runs/demo

## What's inside

- `.claude/skills/` — three Claude skills (md-run-workflow,
  md-prep-structure, md-visualize). Claude Code finds them automatically
  when you work in this directory.
- `run_configs/*.json` — example configs you can copy + modify.
- `tutorial/getting_started.ipynb` — runnable walkthrough.

## With Claude Code

Open a Claude session in this directory and ask:

> "Set up lysozyme in water and minimize it."

Claude routes through the `md-run-workflow` skill, runs the pipeline,
and surfaces the REPORT.md. No CLI knowledge required.
```

### 2.3 Example configs (3)

  - `lysozyme_tutorial.json`: full canonical pipeline (~1 ns production).
    Mirrors the GROMACS tutorial outputs. Long run (~hour).
  - `lysozyme_short.json`: 1000-step EM + 1000-step NVT + 1000-step NPT
    + production disabled. ~90 s end-to-end. The default for `verify.sh`
    to keep the smoke-test fast.
  - `general_md_prep_example.json`: same as `lysozyme_short.json` but
    `pipeline_mode: "general_md_prep"` with `-inter` per-residue
    protonation prompts.

### 2.4 verify.sh

```bash
#!/usr/bin/env bash
set -e
mdagent --version
mdagent doctor --gmx-required --check-network --json | jq -e .ok
mdagent self-test resources --json | jq -e .ok
echo "✓ mdagent is ready. Try:"
echo "  mdagent run-workflow --runs-root ./runs --config ./run_configs/lysozyme_short.json --run-id demo"
```

Hard dependency on `jq`. Should we ship a Python fallback? Or drop the
`jq` and parse with `python3 -c`?

### 2.5 Tutorial notebook (≤ 10 cells)

1. Markdown: "What this project does."
2. Markdown: install commands (link upstream).
3. Code: `subprocess.run(["mdagent", "--version"])` smoke check.
4. Code: write `run_configs/lysozyme_short.json` to disk.
5. Code: invoke `mdagent run-workflow --runs-root ./runs --config
   run_configs/lysozyme_short.json --run-id tutorial`.
6. Code: `mdagent inspect --run-root ./runs/tutorial` — show the
   step ledger + REPORT.
7. Markdown: how to interpret REPORT.md.
8. Code: load `runs/tutorial/step_10_analysis/analysis.json` (if
   analysis ran; for the short config production is disabled so this
   step is skipped — show the EM convergence instead).
9. Markdown: where to go next (visualize, run a longer simulation,
   try general_md_prep mode).

### 2.6 Open design questions

  1. **`init-project` vs. `starter-kit init`** — should this be a
     top-level subcommand or nested under a `starter-kit` group? I
     lean top-level because it's a "first command a new user types."
  2. **Bundle the starter kit inside the wheel** (and reference it via
     `importlib.resources`) **vs. fetch from GitHub at init time?** I
     prefer bundling: offline-usable, no second source of truth, and
     versioned together with the CLI that wrote it.
  3. **`verify.sh` portability** — Windows users can't run a `.sh`
     directly. Should we also write a `verify.ps1`? Or just document
     `mdagent doctor` as the equivalent?
  4. **Should `init-project` create a git repo** (`git init` + initial
     commit)? Lots of "starter-kit" tools do; mine doesn't yet. Adding
     it removes one manual step but also a footgun if the user
     intended to drop into an existing repo.
  5. **The tutorial uses 1AKI from RCSB** — if there's no internet at
     verify time, the smoke test fails. Should the starter kit
     **bundle a sample PDB locally** to make the first run offline-OK?
  6. **`.gitignore` content** — at minimum `runs/*` but not `.gitkeep`.
     Also `__pycache__`, `.venv`? The user may not be using Python at
     all in their MD-study repo. Tighter is fine.
  7. **Tutorial format** — Jupyter `.ipynb` requires `jupyter` or
     equivalent. Should we ship a `tutorial.md` (Markdown) as the
     primary form and the notebook as a generated alternative?
  8. **The 'reproduce this' instruction in the task** — what about the
     case where `mdagent` isn't installed yet on the destination
     machine? Should `init-project` print a one-liner install hint at
     the top of README.md, or attempt to install it?
  9. **What happens if the user runs `init-project DIR` twice?** Refuse
     unless `--force`, or merge non-destructively? I lean refuse-unless-
     force; safer.
  10. **Skill version pin** — the materialized `.claude/skills/` carries
      the skill files as they exist in *this* mdagent version. If the
      user later upgrades mdagent globally but doesn't re-run
      `install-skills`, they have stale skills. Should `init-project`
      copy the skill files **with a comment** indicating which mdagent
      version they came from, plus a one-liner refresh command?
