1. **Default `verify.sh` still breaks the offline promise.**  
WHAT: The script runs `mdagent doctor --gmx-required --check-network` unconditionally. That means the bundled `structures/1aki.pdb` does not actually make `./verify.sh` offline-safe. It also makes the “structural mode” wheel test fail on machines without GROMACS.  
WHY: You explicitly changed the design to avoid network dependence, but the first verification command still depends on network.  
WHAT TO DO: Default `./verify.sh` should do structural/config/skill checks only. Put GROMACS under `--run-smoke`, and make network a separate `--check-network` option used only for the RCSB config.

2. **Do not put a generated comment before skill frontmatter.**  
WHAT: The existing skills start with YAML frontmatter at byte 0. Adding `# Generated by...` at the top may make Claude Code fail to parse/discover the skill.  
WHY: This directly threatens the starter kit’s main purpose: project-local skills work immediately.  
WHAT TO DO: Put generation metadata inside the existing `metadata:` block, or add a comment after the closing `---`, not before the opening `---`.

3. **`MANIFEST.json` has a self-reference trap.**  
WHAT: You say the manifest covers every file and includes hashes. If it includes `MANIFEST.json`, its own hash is unstable/impossible unless specially defined. If it excludes itself, the test wording is false.  
WHY: This will create annoying implementation/test churn.  
WHAT TO DO: Define `files` as “materialized payload files excluding MANIFEST.json”, then separately verify that `MANIFEST.json` exists and has `manifest_schema_version`.

4. **Relative `structure_path` is still cwd-dependent.**  
WHAT: Current ingest uses `Path(structure_path)` as-is, so `./structures/1aki.pdb` works only when the command is run from the project root.  
WHY: `mdagent run-workflow --config /path/to/project/run_configs/lysozyme_short.json` from another cwd will fail.  
WHAT TO DO: Prefer fixing `RunConfig.from_file`/workflow to resolve relative `input.structure_path` against the config file’s directory. If that is too large for v0, document “run from project root” explicitly in README and tutorial.

5. **No inline JSON comment field unless the schema allows it.**  
WHAT: `run_config.schema.json` has `additionalProperties: false`. A comment field in `general_md_prep_example.json` will make the config invalid.  
WHY: The starter kit should not ship an invalid example.  
WHAT TO DO: Put the explanation in README/tutorial, or add a schema-supported `description`/`notes` field intentionally.

6. **The README upgrade command references an unplanned flag.**  
WHAT: You mention `mdagent install-skills --project . --force`, but the current/refactored contract only mentions `dry_run`, not `force`.  
WHY: Users following the README will hit an argparse error unless you add that flag.  
WHAT TO DO: Either add `--force` to `install-skills`, or document the actual refresh command.

7. **C1: bundling `1aki.pdb` is fine, with attribution.**  
WHAT: RCSB says PDB archive data files are under CC0 1.0, and programmatic API data uses the same license; they encourage attribution and warn there is no warranty. The separate RCSB website text/images have different terms, but a PDB coordinate file is the safe case. Sources: https://www.rcsb.org/pages/usage-policy and https://www.rcsb.org/pages/policies.  
WHY: The licensing risk is low; the bigger risk is scientific provenance/version drift.  
WHAT TO DO: Include a tiny `structures/README.md` or manifest metadata: source URL, PDB ID, download date, DOI `https://doi.org/10.2210/pdb1AKI/pdb`, and note that `lysozyme_rcsb_tutorial.json` fetches the live archive copy.
