## Section 1 — Per-issue acknowledgments

**R1-1** (build in-place is wrong — installed wheels are read-only).
**Accept — critical.** Build never writes inside the package. Signature
becomes:

```
mdagent tutorials build --source DIR --out DIR [--pdf] [--notebooks]
```

For users: they call `mdagent tutorials extract DIR` first (which
copies package resources into DIR), then `mdagent tutorials build
--source DIR --out DIR` if they edit. For developers: same command
with `--source src/mdagent/_resources/tutorials` and `--out
src/mdagent/_resources/tutorials` to regenerate before commit.

**R1-2** (extract/build boundary). **Accept.** `tutorials extract DIR`
is a pure copy. PDFs are generated by a separate explicit call after
extraction:

```
mdagent tutorials extract DIR              # copies .md + .ipynb + _build + _shared
mdagent tutorials extract DIR --with-pdf   # extract then build PDFs into DIR
```

No package mutation.

**R1-3** (markdown→notebook splitter underspecified). **Accept.**
Authoring contract pinned:

  - **Fenced ` ```python ` → Python code cell** (executed in the
    default Python kernel when the user runs the notebook).
  - **Fenced ` ```bash ` → `%%bash` magic cell** (so it actually runs
    if the user has bash; ipykernel handles `%%bash`).
  - **All other fences (`text`, `json`, `yaml`, no language)** → stay
    in the surrounding markdown cell as-is (markdown cells preserve
    fences).
  - **Markdown directives in HTML comments** drive notebook metadata
    (see R1-9).

The splitter walks the markdown linearly. Markdown content between
two code fences becomes one markdown cell. Each code fence (only
`python` or `bash`) becomes one code cell. Heading levels do **not**
trigger cell splits (so a tutorial can have nested narrative without
fragmentation).

**R1-4** (bash cells need `%%bash` or `!`). **Accept.** Bash fences
become `%%bash`-cell-magic Python cells. The splitter prepends
`%%bash\n` to the cell source so it Just Works in ipykernel.

**R1-5** (PDFs absent from wheel conflicts with "shipping ready").
**Accept.** Splitting the user-facing story:

  - **Wheel ships:** the canonical `.md` files + the regenerated
    `.ipynb` files + `_shared/pdf.css` + `_build/build.py`. PDFs do
    NOT ship in the wheel.
  - **Release artifacts (GitHub Releases) ship:** a `mdagent-tutorials-v0.1.0.zip`
    bundle that includes pre-generated PDFs. Built by `mdagent
    tutorials build --pdf` against the release commit. Users who want
    the offline-PDF experience download the release asset; users who
    install via `uv tool` get the md+ipynb and generate PDFs locally
    if they install the extra.

In documentation: `mdagent tutorials extract DIR --with-pdf` is the
"give me the full kit including PDFs" command; requires `mdagent[tutorials]`
extra installed.

**R1-6** (`uv tool install --with` syntax). **Accept.** Verifying:
the correct syntax for adding a package's optional extras under
`uv tool install` is via the standard PEP 508 / pip-style
`<spec>[extra]` notation, NOT `--with`. Documentation pin:

```bash
# Install with the tutorials extra (correct form):
uv tool install --force "git+https://github.com/mjayadharan/MDSimulationAgent.git@v0.1.0#egg=mdagent[tutorials]"
# OR (simpler form when uv understands the git URL):
uv tool install --force "mdagent[tutorials] @ git+https://github.com/mjayadharan/MDSimulationAgent@v0.1.0"
```

The README ships a working line, tested. `--with tutorials` is for
"install this OTHER package alongside" — not extras of the primary.

**R1-7** (WeasyPrint isn't dep-free). **Accept.** Documentation
states: "WeasyPrint installs platform-native font/rendering libs via
its wheels on macOS / Linux x86_64 / aarch64. If PDF generation fails,
the notebooks + markdown still work — the PDF is a convenience."
`mdagent tutorials build --pdf` catches WeasyPrint import / render
errors and prints the specific error + a one-line suggestion (e.g.
"on aarch64 macOS, try `pip install --upgrade weasyprint pydyf`").

**R1-8** (PDF determinism). **Accept.** WeasyPrint version pinned in
`[project.optional-dependencies] tutorials = ["weasyprint==X.Y", ...]`.
CSS uses explicit font stacks (`font-family: -apple-system, "Segoe UI",
"Helvetica Neue", Arial, sans-serif;`) so installed-font drift doesn't
break pagination wildly. PDFs are explicitly labeled as "convenience
artifacts; rebuild for your platform; cite the markdown instead."

**R1-9** (markdown directives for notebook metadata). **Accept.**
Authoring directives (HTML comments, ignored by markdown renderers
and PDF):

  - `<!-- mdagent:cell-tag <tag> -->` — adds a tag to the next cell
    (used for "hide-input", "skip-execute", "requires-gmx" etc.).
  - `<!-- mdagent:kernel <name> -->` — sets the notebook kernel
    (default `python3`).
  - `<!-- mdagent:requires <req> -->` — front-matter that adds a
    badge / preface line; aggregated into the notebook's first cell.

**R1-10** (empty-output notebooks need a preface). **Accept.** Every
generated notebook starts with a preface markdown cell:

> **About this notebook:** Cells are shipped without output.
> Open in Jupyter and run them locally — many commands require
> GROMACS, PROPKA, or VMD on PATH. See `<filename>.md` for the same
> content with explanatory prose.

**R1-13** (extract + init-project composability). **Accept.** README
documents the order:

```bash
mdagent init-project ./my-md-study             # scaffold (skills, configs, runs/, structures/)
cd my-md-study
mdagent tutorials extract ./tutorials/         # add tutorial content
```

`mdagent tutorials extract DIR` refuses to overwrite without
`--force`. If `DIR == .` and `.claude/` or `run_configs/` already
exist there (i.e. user ran init-project first), extract just adds
its own files into a subdirectory it owns (the tutorial files
themselves) — no collision.

**R1-14** (build-pipeline tests missing). **Accept.** Adding
`tests/test_tutorials.py` covering:

  - `mdagent tutorials extract DIR` materializes every expected file.
  - For each generated `.ipynb`: `nbformat.validate(...)` passes.
  - For each `.ipynb`: every code cell's source is valid Python
    (compileable; `%%bash` cells are recognized as valid magics by
    ipykernel's parser).
  - `--with-pdf` end-to-end (slow + skipped if weasyprint not
    installed): one tutorial built into a real PDF, asserted
    non-empty + has `%PDF-` header.
  - Build is idempotent: running it twice yields byte-identical
    notebooks (modulo timestamps in cell metadata, which we strip).

**R1-15** (load-bearing: Q1 PDF gen, Q7 build location, Q10 old
notebook). **Accept.** Pinned: WeasyPrint; build operates on explicit
source/out dirs (not in-place); old notebook deleted.
